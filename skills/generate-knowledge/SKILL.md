# Generate Knowledge

分析代码仓库，生成面向 AI Agent 消费的结构化知识库。

知识库的结构规范（文件类型、命名规则、内容模板、质量标准）见 `references/knowledge-spec.md`。本文档定义生成流程。

## 输入

执行前需确认：

| 参数 | 说明 | 必填 |
|------|------|------|
| 代码仓库路径 | 一个或多个 Git 仓库 | 是 |
| 输出目录 | knowledge 文件的写入位置 | 是 |
| 背景文档 | 业务背景描述（如有） | 否 |
| 参考文档 | 已有的分析文档路径（如有） | 否 |

如果提供了参考文档，优先读取作为业务理解的骨架，再用代码验证和补充。参考文档是输入，不是复制源。

## 生成流程

### 第一步：建立背景

1. 读取背景文档（如有），理解业务定位和上下游关系
2. 读取参考文档（如有），建立初步的系统认知

如果没有任何背景信息，直接进入代码分析。

### 第二步：分析代码

对每个仓库，按以下顺序：

1. **入口扫描**：读 CLAUDE.md / README.md，了解定位和技术栈
2. **目录结构**：`ls` 顶层和关键子目录，识别分层架构
3. **核心模块识别**：找到核心业务模块和公共组件，忽略测试/配置/脚手架
4. **业务流程追踪**：从入口（Controller、事件处理器、定时任务）出发，追踪关键调用链
5. **数据模型**：识别核心实体/表及其关系

分析时关注**代码中不显而易见的东西**：
- 为什么这样设计（设计决策、权衡）
- 模块之间怎么协作（跨模块调用关系）
- 哪些规则是隐含的（约定、特殊处理、历史原因）
- 哪些地方容易踩坑（边界情况、已知问题）

### 第三步：规划文件

分析完成后，**先规划再写**：

1. 列出计划生成的文件清单（文件名 + 一句话主题）
2. 确认每个文件的预期行数（overview < 150, index < 100, topic 80-200）
3. 检查文件总数（控制在 5-10 个 topic）
4. 向用户确认文件规划，再开始写

### 第四步：生成文件

按 `references/knowledge-spec.md` 中的模板和规则，依次生成：

1. **overview.md** — 系统地图（< 150 行）
2. **topic 文件** — 各 `.module.md` 和 `.flow.md`（各 80-200 行）
3. **index.md** — 路由表（< 100 行，最后生成，因为需要引用所有 topic 文件名和行数）

index.md 注意事项：
- 用 `git rev-parse --short HEAD` 获取各仓库 commit hash，写在 HTML 注释中作为生成元数据（供人判断时效性，不是给 Agent 校验的）
- 多仓库时在问题路由表中加"仓库"列，标注每个 topic 对应的仓库（跨仓库的列出所有相关仓库）

## 探索范围约束

当仓库与其他文件（如 context、docs）在同一父目录下时，严格控制探索范围：

- **只进入指定的仓库目录**，不对父目录做 `Glob("**/*.md")` 等宽泛搜索
- 背景文档和参考文档通过明确路径读取，不靠搜索发现

## 输出语言

默认中文。代码引用（类名、方法名、文件路径）保持原文。用户可指定其他语言。
